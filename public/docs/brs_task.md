# 1. Overview

The Task Kernel is a **minimal, domain-agnostic execution primitive** used to represent units of work that require action from a human or system.

It is NOT a workflow system, NOT a business system, and NOT a task management application.

It exists solely to provide a **unified execution layer** for all enterprise systems.

It is used by:

* PWM (Process Workflow Management)
* TOS (Team Offering Service)
* HRM (Human Resource Management)
* WFM (Workforce Management)
* EAM (Enterprise Asset Management)
* LMS (Learning Management System)
* DMS (Digital Asset Management System)

---

# 2. Objectives

The Task Kernel aims to:

* Provide a unified task representation across all systems
* Enable assignment of work to actors (users or systems)
* Track execution status of work items
* Maintain traceability to originating system events
* Support lightweight task lifecycle management
* Avoid duplication of workflow or business logic

---

# 3. Core Concept

A Task is a **unit of required action** generated by a source system (usually PWM or domain services).

```text
Source Event → Task Creation → Assignment → Execution → Completion
```

---

# 4. Scope

The Task Kernel covers:

* Task creation
* Task assignment
* Task status tracking
* Task metadata storage
* Task completion logging

---

The Task Kernel does NOT cover:

* Workflow definitions
* State machines
* Approval logic
* Business rules
* Scheduling engines
* Domain logic

# 6. Core Model

## 6.1 Task

A Task represents a single actionable unit.

Attributes:

* title
* description
* source_type (PWM, TOS, HRM, etc.)
* source_id
* entity_type (optional business reference)
* entity_id
* status
* priority
* due_date
* assignment_group (job role)
* assignee (optional staff)

---

## 6.2 Task Status

A Task has a simple lifecycle:

* Queued
* Active
* Done

### Status Rules

* **Queued** → default state when task is created
* **Active** → when assignee is populated
* **Done** → when task is completed

---

## 6.3 Task Assignment Model

Tasks MUST be assigned to an **assignment group (Job Role)**.

Optionally, tasks MAY be assigned to a specific staff member.

### Assignment Rules

* assignment_group is mandatory
* assignee is optional
* task becomes Active when assignee is set

```text
assignment_group = Job Role (required)
assignee = Staff (optional)
```

---

## 6.4 Task Interaction Model

Tasks are **actionable items**.

* Tasks are visible to assignment group members
* If assignee exists, task is owned by that staff
* Task is clickable only by lead assignee or assigned staff
* Clicking a task opens the required action interface

---

## 6.5 Task Metadata

Tasks may include optional metadata:

* JSON payload
* context data
* references to documents/assets
* PWM state reference

---

# 7. Task Lifecycle

```text
Task Created → Queued
Assignment Set → Active
Work Completed → Done
```

Tasks do not control process flow.
They only reflect execution state.

# 9. Business Rules

| Rule ID | Rule                                                          |
| ------- | ------------------------------------------------------------- |
| BR-001  | Task Kernel does not define workflows                         |
| BR-002  | Task Kernel does not contain business logic                   |
| BR-003  | Tasks must originate from a source system event               |
| BR-004  | Each task must reference its origin (source_type + source_id) |
| BR-005  | Task status is Queued by default                              |
| BR-006  | Task becomes Active when assignee is set                      |
| BR-007  | Task becomes Done when completed                              |
| BR-008  | Assignment group (Job Role) is mandatory                      |
| BR-009  | Tasks are actionable via assignment group or assignee         |

# 15. Key Design Principle

The Task Kernel is NOT a system.

It is:

```text
A minimal, unified execution primitive that represents actionable work across all enterprise systems
```

---

# 16. Success Criteria

The system is successful when:

* all systems emit tasks consistently
* no system implements its own task engine
* execution tracking is unified
* workflow engines remain decoupled

---

# 17. Conclusion

The Task Kernel provides a **single execution abstraction layer** across the ecosystem.

It ensures that all actionable work—regardless of source system—is represented consistently and executed uniformly without introducing workflow or business logic complexity.
